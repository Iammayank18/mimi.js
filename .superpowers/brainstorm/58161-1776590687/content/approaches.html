3 Architectural Approaches for mimijs v2

Based on your choices: Dual API · Lifecycle Hooks · Performance · Schema-agnostic Validation

A

Incremental Extension — Lower risk, faster to ship

Keep the current layer stack router. Bolt ctx, lifecycle hooks, and validation on top as wrappers. Radix trie is an optional future upgrade.

Request pipeline — Approach A
http.IncomingMessage
  ↓ prototype-patched req/res (existing)
  ↓ Layer stack scan (O(n) — same as today)
  ↓ ctx wrapper built per-request
  ↓ hook slots injected as special layers
  ↓ ValidatorAdapter.parse()
  ↓ handler (ctx or req/res detected by arity)
finalhandler

Wins

  • Preserves all current work
  • Lowest implementation risk
  • Ship hooks + validation fast

Costs

  • Layer stack = O(n) routing — performance ceiling hit early
  • ctx built on top of req/res = double allocation per request
  • Hooks feel bolted on, not native
B

ctx-First Core + Express Compatibility Adapter ★ Recommended

New lean core: radix trie router + named lifecycle hook slots + a single RequestContext per request. Express req/res/next is a thin adapter. One coherent architecture, not a wrapper on a wrapper.

Request pipeline — Approach B
http.IncomingMessage
  ↓ RequestContext created (one object, typed)
  ↓ Radix trie match (O(log n))
  ↓ hook: onRequest
  ↓ hook: preParsing → body parsed lazily
  ↓ hook: preValidation → ValidatorAdapter.parse()
  ↓ hook: preHandler
  ↓ handler(ctx) or handler(req,res,next) ← adapter
  ↓ hook: onSend → schema serialization
  ↓ hook: onResponse
onError hook → error handler

Wins

  • Radix trie = O(log n) routing, 2–3× faster under load
  • Schema serialization at onSend — fastest JSON path
  • Hooks are first-class, not middleware hacks
  • TypeScript generics flow through ctx naturally
  • Express adapter is ~100 lines, well-understood pattern

Costs

  • More implementation work than A
  • Compatibility adapter needs careful testing
C

Micro-kernel + Everything-is-a-Plugin — Maximum flexibility

The core is ~200 lines: HTTP server + hook registry + plugin system. Router, ctx, validation, body parsing are all built-in plugins. Third-party plugins get their own scope — routes and hooks can't leak to parent.

Architecture — Approach C
kernel: hookRegistry + pluginLoader + http.Server
  ↓ registers built-in plugins at boot:
    @core/router (radix trie)
    @core/context (RequestContext)
    @core/body-parser
    @core/validator
    @compat/express (opt-in)
  ↓ user plugins register into scoped containers
  ↓ request flows through scope chain

Wins

  • Most "adaptable" — swap any built-in for custom
  • Plugin encapsulation prevents interference
  • Future-proof for multi-runtime (Bun, Deno)

Costs

  • Highest complexity — months more implementation
  • Harder to document and onboard
  • Indirection makes debugging harder

I recommend B — best balance of performance, clean architecture, and Express compatibility. Pick one.
